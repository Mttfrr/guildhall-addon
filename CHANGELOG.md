# Changelog

All notable changes to GuildHall will be documented in this file.

## [0.7.4-beta] — Unreleased

### What's New

- **Planned vs actual raid-comp diff.** When you're mid-raid AND viewing the planned comp for the event being tracked, a diff strip appears below the Group 1-8 list: `Planned 25 · In raid 22 · Present 21` plus `Missing: Alice (G1), Bob (G3)` (sorted by group) and `Subbed in: Charlie`. Catches "who hasn't joined yet?" + "who got swapped in without me updating the platform comp?" in a single glance. Data was already on both sides (raidComps for the plan, currentSession.members for the in-flight roster); just no view that connected them.
- **Class-balance warnings on the planned comp.** Yellow strip above the Group 1-8 list when the comp has obvious balance issues: `⚠ No combat rez (DK / Druid / Hunter / Warlock) · No Bloodlust (Shaman / Mage / Hunter / Evoker) · 4x Mage stacking`. Tank/healer count warnings gate on comp size ≥18 so heroic / M+ comps don't get spammed. Empty result = comp looks balanced, render nothing.
- **Raid Comp section groups by raid group (1-8) instead of by role.** Matches the platform's Raid Comp builder. Within each group, members sort tanks → healers → DPS for at-a-glance scanning, with a per-name role badge (T/H/D in colour). "Unassigned" tail bucket for slots without a group set. Auto-sort after `/gh invite` now fires whenever the comp has any group assignment (was previously gated on the unlikely "invite source was comp" case, missing the common signups-driven flow).
- **Officer "Mark status" right-click on Roster rows.** Right-click any signed-up player → "Mark status ▸" submenu with all 8 status codes (Present / Late / Late-officer / Bench / Tentative / Absent / Left-early / Replaced-mid-raid). Mutates `db.global.signups` in place AND queues into `db.global.pendingSignupChanges` for export. Platform's import endpoint applies the queued changes idempotently and the queue clears on the next successful import.
- **Stale-data banner across the top of the main frame** when `db.global.lastImport > 7 days ago`. One yellow line + a "Sync now" button that jumps to the Sync tab. Re-evaluates on tab switch, OnShow, and after every import.
- **"What's new" badge in the title bar** when the running version is newer than `db.profile.lastSeenVersion`. Click → opens the modal in place. No login auto-pop; the badge stays opt-in until you click "Got it".
- **`/gh search <name>`** — cross-context character lookup. Walks loot / attendance / signups / teams / wishlists and prints counts + sample rows. Replaces the four-different-tabs-and-eyeballing flow with one chat dump.
- **`/gh export <table>`** — selective single-table export with a copy-via-EditBox popup. Useful for the "I just want to update the loot ledger" workflow when the full export is large or partially stale. Allowed: `loot`, `attendance`, `encounters`, `raidCompResults`, `guildBankMoneyChanges`, `guildBankTransactions`.
- **`/gh diag`** — db.global health summary. Addon version, last-import / last-export age, current-team filter, row counts for every telemetry and imported table, in-flight session state. Telemetry rows highlight orange when they cross "unusually large" thresholds.
- **CI quality gates.** Three new automated checks: TOC ↔ `WGS.version` consistency (catches release-bump drift before merge), removed-API scan (curated list of WoW APIs removed at TOC 110000+ — would have caught the EasyMenu break at PR time), and a public-event contract spec against `docs/EVENTS.md` (drift between the documented payload and the actual fired payload becomes a CI failure). Also a UI shim layer in `spec/helpers.lua` that mocks `CreateFrame` + `MenuUtil` + `StaticPopupDialogs` so menu builders can be unit-tested. **295 specs total** — up from ~250.

### Changed

- **All five right-click menus migrated from `EasyMenu` to `MenuUtil.CreateContextMenu`.** EasyMenu was removed in retail's 11.0 interface (TOC 110000+) and every right-click in the addon was erroring on current retail. New `ui.OpenContextMenu(menu)` compatibility helper preserves the existing menu-table format so per-callsite construction code didn't have to change.
- **Export envelope auto-normalizes malformed attendance scalars at the source.** Type-wrong fields (a Lua table where a number was expected, observed in the wild) get nil'd in place before serialisation. One-line `Normalized N malformed fields…` print so the user knows their data was self-healed. The platform schema stays strict so genuine corruption surfaces as a clear rejection.
- **Release workflow grants `contents: write` to `GITHUB_TOKEN`** so the BigWigsMods packager can update the GitHub release at the tag. Default token is read-only on modern repos; without this, every release run died with a late-stage 403 on the release PATCH.
- **`WGS.version` in `Core.lua` synced to `0.7.3`** — had silently lagged at `"0.7.0-beta"` across two version bumps because only the TOC was bumped. The new TOC ↔ Core consistency check (above) prevents future drift.
- **`WGS:AutoInvite` accepts an optional event override** so the rail kebab's "Invite signups" targets a specific event. The no-arg `/gh invite` call site is unchanged.
- **Client-side decode errors include per-stage diagnostics** instead of the previous catch-all "Export string appears truncated." Now: checksum mismatch lists expected vs computed hex + payload length, decode-character errors point at the offending character, decompression failures carry context. Server validation errors include the Zod path + per-field issue message instead of just "Validation failed".

### Removed

- **`EasyMenu` dependency** — gone, replaced by `MenuUtil`. Legacy `_G.GuildHall*Dropdown` per-menu frame instances dropped (no longer needed under `MenuUtil`'s lifecycle).
- **Login auto-pop of the "What's New" modal.** Auto-popping a modal on every login after an update felt intrusive — surfaced as a title-bar badge instead. Opt-in, click to open. `PLAYER_ENTERING_WORLD` no longer touches the modal.
- **The `.catch(null)` schema loosening on `addonAttendanceSessionSchema`** that was added in a previous attempt. Restored the strict guardrail; bad addon-side data gets normalized at the source on export (above) instead. Same end state, principled architecture.
- **The strict `source == "raid comp"` gate on auto-sort post-invite** — replaced with "fires whenever the event has any group assignment." Signup-driven invite flows (the common case) now place raiders in their planned groups too.

## [0.7.3] — 2026-05-26

### Added
- **Right-click context menu on player names.** Single shared menu (`Whisper · Invite · Inspect · Copy name · Copy profile link`) wired to every player-name surface the addon renders. Right-click in three places this release: Logs → Attendance member grid, Events Roster section row, Logs → Loot row's player cell. Loot row right-click splits intelligently — clicking on the player cell opens the player menu; clicking elsewhere on the row keeps opening the existing loot row menu (Re-tag event / Delete row). The "Copy profile link" item is conditional: only renders when the platform-side `characterIds` map (new in the paired addon-sync export change) has a memberId for the clicked name. Pops the same auto-selected EditBox popup the other "Copy" affordances use — one Ctrl+C copies, paste anywhere.
- **Right-click kebab on event rail rows.** Each event in the Events tab's left rail gets a right-click menu with "Copy event link" and "Invite signups." Event link uses the EditBox-popup workaround for Blizzard's no-browser-launch sandbox restriction. Invite signups calls `WGS:AutoInvite` against the right-clicked event specifically (the function gained an optional `eventOverride` argument; the no-arg /gh invite call site behaves the same as before).
- **Right-click quick-action menu on the minimap button.** Replaces the previous "right-click → /gh config" shortcut. New menu: `Show / hide main frame · Sync now · Start/Stop attendance tracking · Open Sync tab · Settings…` Matches the WoW addon convention every major addon (DBM, WeakAuras, BigWigs) follows for minimap button affordances. Tooltip's "Right-click" hint updated to "Quick actions menu."
- **Roster section gets a filter-by-name search box** on the Events detail panel. Lifts the search-input pattern from Logs → Loot. At ≥20-person raid scale, finding "did Bob actually sign up?" no longer means scrolling + eyeballing the full list. Case-insensitive substring match; Esc clears; empty buckets drop out of the render so the panel doesn't show "Tentative (0)" headers under a tight filter.
- **Status-pill hover tooltips** on the Events rail. `Today / Soon / Upcoming / Past` were opaque labels if you didn't know the cutoffs; the tooltip turns each into a self-documenting affordance ("Soon = within 7 days", "Past = scheduled start was >3h ago", etc.). Hover only over the pill area, not the whole row — click forwarding lets the pill still toggle expansion.
- **Chat lines on attendance start + stop.** Matches the `Bank captured: …` confirmation style — silent capture lifecycle is over. Start: `Attendance tracking started: Team Alpha · The Voidspire`. Stop: `Attendance recorded: 25 members · 2h 14m · 3 boss kills. /gh attendance to review.` Boss kill suffix only renders if MRT/NSRT contributed `bossAttendance`. Fires regardless of how the session started (auto on RAID_INSTANCE_WELCOME, manual via Track button or shift-minimap).

### Changed
- **`/reload` mid-raid no longer drops the in-flight attendance session.** Previously the session lived in a module-local that reset to nil on every addon load, while finalized rows only land in `db.global.attendance` via `StopAttendance`. So `/reload` between start and stop abandoned everything captured up to that point. Fix: `StartAttendanceForTeam` aliases the same Lua table into `db.global.activeSession`; mutations write through to SavedVariables via reference semantics for free. `StopAttendance` clears the alias. `OnEnable` calls `rehydrateActiveSession` on load and restores the local from SavedVariables, with an 8h orphan guard so a session abandoned for the night (logged off mid-raid, came back next day) gets quietly dropped instead of resurrected.
- **`WGS:AutoInvite` accepts an optional event override.** Lets the rail kebab's "Invite signups" target a specific event. The no-arg `/gh invite` flow falls back to today's event resolution exactly like before.
- **Platform addon-sync export ships a `characterIds` map** (paired platform change). Flat `{ [character_name]: guild_members.id }` lookup that the addon imports into `db.global.characterIds` for the player context menu's profile-link feature. Older addon versions ignore the new field harmlessly; older imports on the addon side gracefully hide the "Copy profile link" menu item.

## [0.7.2] — 2026-05-26

### Fixed
- **Cross-timezone event status (Paris raid viewed from HK shows "Past" mid-raid).** Guilds with members across regions saw events flip to "Past" — and the Track button disable — long before the raid actually ended. The addon's event-time helpers parsed `date + time` via Lua's `time(table)`, which interprets the fields in the user's local timezone — so a Paris 20:00 raid viewed from Hong Kong got parsed as 20:00 HK = 14:00 Paris, 6h before the real start. At 22:30 Paris (mid-raid) the computed delta was ~-8.5h, tripping the `delta < -3h → "Past"` rule. Fix: the platform's addon export now ships pre-computed UTC unix seconds (`start_ts`, `end_ts`) per event, derived from the event's IANA timezone via `Intl.DateTimeFormat`. The addon's `ParseEventTime`, `EventStartTs`, `EventStatus`'s "Today" check, and `FindTodayEventForTeam` all prefer the new field, falling back to the legacy local-time parse for events imported before the fix shipped (re-import is the clean upgrade path). As a bonus, the previously-flaky `auto_attendance_spec:88` test now passes consistently — it was actually catching the same root-cause bug, just intermittently around UTC midnight.
- **Track button stays clickable mid-raid even when status reads "Past".** The Track button in the Events detail panel's actions footer was disabled whenever `EventStatus` marked the event as "Past" (which fires at `delta < -3h`, but 3.5-4h raid sessions are normal). Officers running long sessions who needed to manually start tracking — auto-flow missed because of a reload, addon loaded mid-fight, accidental stop — hit a disabled button. New carve-out: when `IsInRaid()` is true, the button stays enabled regardless of the "Past" label. The original guard against creating sessions for last-week's-raid from town still applies when solo.

### Changed
- **Slash command dispatch table replaces the elseif chain in `Core.lua`.** `SlashCommand` had grown to ~25 `elseif cmd == ...` branches with per-command comments interleaved between rungs. Replaced with a `SLASH_HANDLERS` table mapping canonical command → `function(self, input)`, a `SLASH_ALIASES` table for alternate spellings (the `[""] = "show"` entry routes bare `/gh`), and a 6-line dispatcher that falls through to `L["SLASH_HELP"]`. Each command is now self-contained; per-command docs live above their handler entry instead of between control-flow rungs. Hidden dev commands (`interop`, `peerloopback`, `restore`) stay out of `L["SLASH_HELP"]` — the public help string only lists user-facing surface. No behaviour change.
- **Event-picker menu builder shared between Logs → Loot and Logs → Attendance.** `OpenLootRowMenu` (right-click on a loot row) and `OpenSessionEventMenu` (Attendance expand → "Bind to event…") were building the same shape independently: walk `db.global.events`, filter to ±4h of a reference timestamp, sort by closest-in-time, format `MM/DD HH:MM · Team · Title`, append an empty-state placeholder + clear entry at the bottom. Extracted as `BuildEventCandidateItems(refTs, opts)` returning the array of menu items; the loot menu wraps it in a "Re-tag event ▸" submenu, the session menu renders it flat under a "Bind to event" title. The next correction surface that wants an event picker reuses the helper instead of copy-pasting.
- **Correction mutators share a hint + cascade helper.** The "Local change saved. Other officers will need to apply the same correction or re-import from the platform." string was duplicated as a const in both `Modules/Loot.lua` and `Modules/Attendance.lua`. Lifted to `WGS:PrintCorrectionHint()` on Core; single source of truth for the day cross-officer edit propagation lands (one line to update). Three Attendance mutators (`RebindAttendanceSession`, `RemoveMemberFromSession`, `DeleteAttendanceSession`) each open-coded "walk raidCompResults backwards, find rows matching this session's startedAt, mutate" — extracted as a `forEachCompSnapshotAt(startedAt, fn)` helper that walks comps in reverse so `fn` can `table.remove` safely. Each mutator's cascade site reads ~3 lines shorter.

### Added
- **`/gh peerloopback` — dev-only PeerSync loopback mode for single-client testing.** Officer-sync testing normally needs a second WoW client logged in as another officer in the same guild — inconvenient and noisy on real guild channels. New profile flag (`db.profile.peerSyncLoopback`, default false) re-feeds every outbound chunk through the local dispatch path with our own playerKey as senderKey and `isSelf` forced false. Exercises the full encode → decode → trust gate → merge fn pipeline AND the catch-up handshake chain (`__probe → __offer → __request → table replay`) from a single client. Toggle via `/gh peerloopback`. Kept out of the public help string since this is a developer-facing affordance. Doesn't test real-network packet-loss / chunk-reorder / GUILD throttle — those still need a second client — but covers ~95% of the surface from a solo character.

## [0.7.0-beta] — 2026-05-24

### Added
- **In-addon loot + attendance corrections.** When a captured row is wrong, the previous-only fix was export → edit on the platform → re-import. Officers regularly hit this — auto-resolve picks the wrong event, a test-pull session needs deleting, a mid-pull join didn't get captured. The data model already carried every field needed to fix in place; the UI just didn't expose it. New surfaces, all routing through a shared mutator API on `Modules/Loot.lua` + `Modules/Attendance.lua` (so the UI never touches `db.global.*` directly):
  - **Logs → Loot, right-click a row** → context menu with "Re-tag event ▸" (submenu of every event in `db.global.events` within ±4 h of the row's timestamp, sorted by closest-in-time, each labelled `MM/DD HH:MM · TeamName · EventTitle`, plus an "Untag" option at the bottom), and "Delete row" (no confirm — recoverable via MRT gap-fill if the row was recent).
  - **Logs → Attendance, click a session to expand** → the expanded body gains an actions row with "Bind to event…" (same ±4 h menu shape, single-level, no submenu — plus "Clear binding"), "Delete session" (gated behind a `StaticPopup` confirmation because the cascade into `raidCompResults` is irreversible), and a per-member 14×14 "x" button on each name in the 3-column grid (turns red on hover; click → splices the member out of `memberList` AND out of every matching `raidCompResults.slots[]` so the per-pull comp stays consistent with the corrected roster).
  - **Cascade semantics.** `RebindAttendanceSession` / `RemoveMemberFromSession` / `DeleteAttendanceSession` all walk `db.global.raidCompResults` and update every row sharing the session's `startedAt`. Otherwise removed members would still appear "present" in the per-pull comp, and rebound sessions would have orphan snapshots pointing to the wrong event.
  - **Cross-officer propagation deliberately out of scope for v1.** PeerSync's per-row merge is first-wins on `(itemID, shortName(player), timestamp ±60s)` for loot and `(startedAt, startedBy)` for attendance, so edits to rows another officer already has get dropped as dupes. Each mutator prints a `"Local change saved. Other officers will need to apply the same correction or re-import from the platform."` hint so the limitation stays visible. A future commit can add edit-broadcast (per-row `rev` counter + LWW); cleaner to ship the in-addon edit surface first.
  - 18 new specs (`spec/loot_corrections_spec.lua`, `spec/attendance_corrections_spec.lua`) lock down the happy paths, the cascade into `raidCompResults`, the out-of-range guards, and the local-hint print contract.
- **`/gh interop` — MRT/NSRT integration diagnostic.** Before this, the only way to verify the bridge code was actually doing anything was to `/dump WGS.db.global.loot` and grep for `source = "mrt"`, then `/dump WGS.db.global.attendance` and check the `bossAttendance` field on the latest session — fine for a developer, not a feature. New slash command prints a structured status: MRT loaded yes/no, NSRT loaded yes/no, VMRT global presence + sub-table count, NSRT global presence, gap-fill loot count (`N rows tagged source=mrt out of M total, last: 2m ago`), session count with `bossAttendance` attached, MRT note size + which public API actually fetched it (`MRT.F.GetNote` vs `GMRT.F:GetNote` vs raw `VMRT.Note.Text1`). Returns a structured table from `WGS:InteropStatus()` so future UI surfaces (a Sync-tab pill, a dashboard health row) can reuse the same data without re-implementing the inventory pass. 3 new specs cover the empty-slate, MRT-loaded-with-data, and NSRT-only-with-VMRT-fallback scenarios.
- **Cross-addon interop contract re-verified against upstream and tightened.** Pulled latest MRT (`akbyrd/method-raid-tools`) and NSRT (`Reloe/NorthernSkyRaidTools`) source, diffed against `docs/INTEROP.md`. Confirmed no breaking drift on any contract GuildHall currently reads (attendance row shape, loot pipe-string format, note text accessor, NSRT nickname map). Tightened four sibling-field gaps: documented `VMRT.Attendance.{enabled, specialEdit, OptionsSortPD}` siblings and the `alts` array shape, added `VMRT.LootHistory.disable` (user-toggleable capture flag), pinned `NSRT.InviteList[listName] = { "Name", ... }` (was "shape not stable"), added `NSAPI:GetAllCharacters / GetName / GetCharacters` as the proper public-read path for nicknames (doc previously only listed the write path). Logged six new surfaces worth knowing about but **not** building bridges for yet: MRT `Marks.list` / `ExCD2` / `RaidCheck` / `WhoPulled`, NSRT `CooldownList` / `ReadyCheck` gear-buff checks. Last-verification date bumped to 2026-05-24.
- **Manual start/stop attendance from the Events tab + minimap.** Attendance capture stays auto-triggered (RAID_INSTANCE_WELCOME → start, GROUP_LEFT → stop), but the addon now exposes a manual override for the cases the auto-flow misses (quick clears that don't fire the welcome event, mid-raid joins, accidental stops). Two surfaces: a fifth button (`Track` / `Stop`) in the Events detail panel's actions footer, scoped to the currently-selected event — starting via this button explicitly binds the session to `ev.team_id`, bypassing the `FindActiveScheduledEvent` auto-resolution; the button greys out on past events when nothing's tracking. And a shift-left-click on the minimap button (tooltip updated to advertise the new chord). Both paths route through `WGS:StartAttendanceAutoTagged()` (minimap, auto-resolved team) or `WGS:StartAttendanceForTeam(ev.team_id, ...)` (Events footer, explicit event). The "Share Gear Gaps" button trims to "Share Gaps" to make room for the new fifth button in the ~480 px footer. Logs → Attendance stays a pure read surface (sessions list).
- **Loot rows now carry their event + team at capture time.** When `CHAT_MSG_LOOT` fires (or the MRT reconciliation gap-fill runs), the captured row is stamped with the active attendance session's `eventId` + `teamId` via a new accessor `WGS:GetCurrentAttendanceContext()`. The Logs → Loot sub-view's team filter switches from a fuzzy date-window heuristic (loot ±5h of any of the picked team's events) to an exact `entry.teamId == currentTeamId` match. Loot captured before this version stays untagged and is excluded from team-filtered views (visible under "All Teams" as before). The platform-side export carries the new fields through verbatim — existing consumers ignore them; new ones can use the exact event link. 3 new specs cover the context accessor's nil / active / post-stop states.
- **Manual officer sync from the Sync tab UI.** The `/gh sync` slash command (peer-sync catch-up — probes other officers for newer captures and pulls deltas) already existed but was invisible unless you knew the slash. Added a dedicated "Officer Sync" section at the top of the Sync tab with a "Sync now" button + a one-line status read-out: enabled/disabled state, current channel (RAID / PARTY / GUILD or "no channel"), last sync timestamp ("3m ago"), and peer count from the last round. The button greys out automatically when the sync can't run (not officer rank, no channel, or a probe is already in flight) and shows a "probing…" indicator while the offer-collection window is open. New public accessor `WGS:PeerSync_Status()` returns the snapshot for any future UI surface; 3 new specs cover the enabled/officer/channel branches.
- **Officer sync now covers events, teams, signups, and the rest of the import payload.** The previous round of officer-sync only covered append-only telemetry (loot, attendance, encounters, raid-comp results) — anything that comes from the platform as a whole-table import was left out, so two officers could be looking at different rosters or schedules if one had imported a fresher dump than the other. New mechanism: each officer's `db.global.lastImport` doubles as a snapshot version, piggybacked onto the existing probe/offer handshake as `__snapshot` stamp. If a peer's stamp is meaningfully newer (more than 30 s ahead, the grace window that absorbs same-dump-different-import-time stamp drift since the platform doesn't issue a dump ID), the local officer sends one `__request` for `table = "__snapshot"`. The targeted peer responds by broadcasting their current import payload (all 11 fields: characters, teams, wishlists, bossNotes, raidComps, events, gearAudit, characterDetails, signups, targetIlvl, serverMinAddonVersion) as a single `__snapshot` row. Receivers replay it through the existing `WGS:ProcessImport(...)` pipeline, so all the normal post-import side effects (BuildCharacterLookup, WGS_IMPORT_APPLIED fire) happen for free. Pull-only by design — there's no broadcast-on-import; sync happens at raid entry or `/gh sync`. The Sync tab status line now shows `import: 2h ago` so officers can see when snapshot catchup has lifted them onto a newer payload (and spot the "the whole guild is on stale data" case). 8 new specs cover the offer-stamp piggyback, the grace-window decision, the request → response flow, and the stale/malformed-payload skip paths. Throughput note: payload chunks at ~10/s on RAID (≈20 s for a typical guild's data), ~0.5/s on GUILD (slow but acceptable for manual sync from town). Limitation: no per-row diff, so receiving a snapshot wholesale-replaces the snapshotted tables — local-only edits to events/teams/signups would be lost, but the addon doesn't currently have any local-edit surfaces for these tables, so this is theoretical today.

### Removed
- **`/gh bankdebug` diagnostic + `bankDebugPrint` scaffolding.** Added during the bank-capture investigation so the user could surface which (if any) of the four bank events were arriving on their client. The auto-flow is now verified working in production (`GUILDBANKLOG_UPDATE` subscription + `PLAYER_INTERACTION_MANAGER_FRAME_SHOW` backup are doing their job), so the trace prints + the `/gh bankdebug` slash branch + the `WGS._BankDebugPrint` export are stripped. The actual fixes stay — `QueryGuildBankLog(MAX_GUILDBANK_TABS + 1)` on `_HandleBankOpened`, the `OnLogUpdate` handler, the `PLAYER_INTERACTION_MANAGER_FRAME_SHOW` subscription, the `WGS_BANK_CAPTURED` event + Bank sub-view live refresh, and the "Capture now" button. Net: 21 lines deleted, zero behavior change in the working state.

### Fixed
- **Raid Comp section cascading indent in the Events detail panel.** Each member row was indenting deeper than the previous one — Teraiz at column 12, Healers heading inheriting that 12, Deconaga at 24, Chaiiteea at 36, until DPS members ended up halfway across the panel. Cause: every FontString was anchored `TOPLEFT → previous element's BOTTOMLEFT` with `+12` x-offset for members and `0` for role headings, so the x-position cascaded down the chain. Fix: split into two anchors per row — TOP relative to the previous element (for vertical stacking), LEFT relative to the section header (stable x column). Now role headings render at `header.x` and members at `header.x + 12`, regardless of what came before.
- **Raid comp "missing for event #n" on platform import.** Two distinct capture-side bugs both surfaced as the same platform error and are both fixed.
  - **Bug B — kill-less sessions sometimes recorded zero snapshots.** `WGS:SnapshotRaidComp` was only called on `ENCOUNTER_END` (one per boss kill) and at session end. `CaptureRaidComposition` filters to members with `present == true and subgroup > 0`, and `OnGroupRosterUpdate` flips `present = false` when a member leaves. So a session that killed no bosses AND emptied out before `StopAttendance` (everyone left to log off) produced two empty snapshot attempts and recorded nothing. Fix: take a snapshot at `StartAttendanceForTeam` time, when the roster is fresh from `GetRaidMembers` and every member has `present = true` — guarantees at least one `raidCompResults` row per session.
  - **Bug A — sessions outside the strict auto-resolve window stayed untagged.** `FindActiveScheduledEvent` uses a `[start − 30 min, start + 1 h]` window to avoid silent mis-tagging. Raids that started outside that window (event scheduled but officers ran late / early, or the event was created on the platform AFTER the raid started and only landed via a later import) got `eventId = nil`, snapshots inherited the nil, and the platform couldn't link them. Fix: at `StopAttendance`, re-run `FindActiveScheduledEvent` with a wider `[start − 2 h, start + 4 h]` window, and if exactly one event matches, backfill `currentSession.eventId/eventTitle/pullTime` AND any `raidCompResults` rows already recorded for this session. Ambiguous matches stay untagged so an officer never has to undo a silently-wrong bind. If still untagged, print a chat hint pointing at the new `/gh attendance reconcile` for after the import lands.
  - **Retro fix for existing orphan data.** New `WGS:ReconcileAttendanceEventBindings()` (exposed via `/gh attendance reconcile`) walks every attendance row whose `eventId` is nil and re-runs the same wider-window matcher against the session's `startedAt`. Backfills the session AND every `raidCompResults` row sharing that timestamp. Reports `(bound, ambiguous, unmatched)` counts so the user can see whether to re-import more data or accept that the unmatched raids genuinely had no scheduled event. `FindActiveScheduledEvent` extended to accept `(now, leadOverride, trailOverride)` so the wider-window path is one signature, not a duplicate function. 9 new specs across `auto_attendance_spec.lua` lock down the start-snapshot guarantee, the back-resolve happy path, the no-match-stays-untagged path, the retro reconcile happy path, the ambiguity disambiguation, and the don't-overwrite-existing-bindings guard.
- **MRT integration now works for NSRT users too.** Every MRT bridge site (loot reconciliation, per-encounter attendance from `VMRT.Attendance.data`, MRT notes) gated on `WGS:HasAddon("MRT")` — so NSRT users (Method's modern fork, which still writes to the `VMRT` global for backwards compat) got `false` from the gate and zero benefit from the integration. New helper `WGS:HasMRTData()` in `Util/Interop.lua` returns true if MRT *or* NSRT is loaded, with a fallback to checking `type(_G.VMRT) == "table"` directly (covers any future addon that writes the same structures). Four call sites swept over. Three pre-existing specs that guarded against "VMRT-populated-but-MRT-not-loaded = ignore" updated to reflect the new contract (`VMRT presence is the signal`), plus 3 new specs locking in the NSRT-compat read path for each bridge.
- **Bank capture, second pass — the real root cause.** The previous round (QueryGuildBankLog + 2 s deferred capture) was necessary but not sufficient: even with the query issued, the addon still didn't read the response. Other addons (or our own future MRT/NSRT bridge modules) can subscribe via `GuildHall.RegisterCallback(self, eventName, handler)`. Initial events: `WGS_SESSION_STARTED` and `WGS_SESSION_ENDED` (attendance), `WGS_IMPORT_APPLIED` (after a successful import merge), `WGS_LOOT_RECORDED` (each loot row). Documented in `docs/EVENTS.md` with payload shapes.
- **/gh invite uses event signups** — `WGS:GetEventInviteList` now picks the invite source in this order: (1) event signups from the web (committed statuses only — P/L/B/LT), (2) raid comp assignments, (3) team roster. Before, signups were ignored and the addon always went straight to comp/roster, which meant officers who used the web sign-up flow were inviting everyone on the team instead of just the people who said they were coming. The web now ships `data.signups` in the export response.
- **Instant invites** — the 3-second-per-invite stagger in `WGS:AutoInvite` is removed. A 25-person raid used to take 75 seconds to fully invite; now it fires in one tick. The post-invite sort-groups still waits 5s for accepts to land.
- **Export envelope v3 with checksum** — addon now emits `WGS3<8-hex-djb2>:<base64>` and the web emits the same format. The decoder validates the djb2 checksum on the base64 string and rejects truncated paste with a clear error ("Export string appears truncated — please re-copy the full string."). The legacy `WGS<base64>` (v2) envelope is still accepted on import for backwards compatibility.
- **Outdated-addon banner** — Dashboard tab now shows a banner when the running addon is older than the server's `MIN_ADDON_VERSION` (read from the web's export response on import). No more raw error strings on next sync.
- **Auto-populated export on Sync tab open** — the post-raid reminder's "Export Now" button now lands on the Sync tab with the export string already generated, selected and ready for Ctrl+C.
- **Snapshot-before-clear safety net** — "Clear exported data" now stashes a copy of loot/attendance/encounters/bank into `db.global.lastClearSnapshot` first. Type `/gh restore` within 24h to undo. Protects against the "I cleared but my paste was actually truncated" failure mode.

### Changed
- **Core.lua slimmed from 587 → 229 LOC** by extracting utilities into a new `Util/` directory: `Util/JSON.lua` (ToJson/FromJson/JSON_NULL), `Util/Base64.lua` (Base64Encode/Base64Decode/HashString), `Util/Time.lua` (GetTimestamp/GetPlayerKey), `Util/Roster.lua` (NormalizeFullName, BuildCharacterLookup, ResolvePlayerForCharacter, GetGuildRosterLookup, IsGuildGroup, CLASS_COLORS).
